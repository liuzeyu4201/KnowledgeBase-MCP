# 知识库 MCP Server 数据库设计方案（第一阶段）

## 1. 设计目标

本方案用于支撑知识库 MCP Server 第一阶段建设，面向以下能力：

- 分类管理
- PDF 文档导入
- 文档切片存储
- 稀疏向量与稠密向量检索
- 文档、切片、向量的增删改查
- 上层 Agent 通过 MCP Server 调用知识库能力

第一阶段采用以下存储分工：

- **关系型数据库**：保存业务主数据
- **Milvus**：保存向量索引数据
- **文件存储**：保存原始 PDF 文件

正式环境推荐：

- 关系型数据库：PostgreSQL
- 向量数据库：Milvus
- 服务实现：Python
- 环境与依赖管理：uv

---

## 2. 实体拆分方案

第一阶段统一采用固定实体建模，不采用“每个 PDF 一张表”的方式。

### 2.1 分类实体 `Category`

用于管理知识分类，作为文档归属的组织维度。

主要属性：

- 分类主键
- 分类编码
- 分类名称
- 分类描述
- 分类状态
- 创建时间
- 更新时间
- 删除时间

---

### 2.2 文档实体 `Document`

用于表示一份 PDF 文件，是文档生命周期管理的核心实体。

主要属性：

- 文档主键
- 文档稳定标识
- 所属分类
- 文档标题
- 文件名
- 文件存储地址
- 文件摘要
- 文件类型
- 文件大小
- 解析状态
- 向量状态
- 文档版本号
- 切片数量
- 错误信息
- 创建时间
- 更新时间
- 删除时间

---

### 2.3 切片实体 `Chunk`

用于表示文档切片后的内容片段，是检索回查的核心实体。

主要属性：

- 切片主键
- 切片稳定标识
- 所属文档
- 切片序号
- 页码
- 原文偏移
- Token 数量
- 切片内容
- 内容摘要
- 向量模型
- 向量版本
- 向量状态
- 扩展元数据
- 创建时间
- 更新时间
- 删除时间

---

### 2.4 向量逻辑实体 `ChunkVector`

用于表示切片在 Milvus 中对应的向量索引记录。

第一阶段设计要求：

- 每个切片对应一条当前有效向量记录
- 一条向量记录同时保存稀疏向量与稠密向量
- 向量记录通过 `chunk_id` 与业务数据关联

---

## 3. 关系型数据库表设计

## 3.1 分类表 `kb_category`

| 字段名 | 类型 | 必填 | 唯一 | 索引 | 字段说明 |
|---|---|---:|---:|---:|---|
| `id` | `BIGSERIAL` | 是 | 是 | 是 | 分类主键 |
| `category_code` | `VARCHAR(64)` | 是 | 是 | 是 | 分类编码 |
| `name` | `VARCHAR(128)` | 是 | 是 | 是 | 分类名称 |
| `description` | `VARCHAR(512)` | 否 | 否 | 否 | 分类描述 |
| `status` | `SMALLINT` | 是 | 否 | 是 | 分类状态，`1` 启用，`0` 停用 |
| `created_at` | `TIMESTAMP` | 是 | 否 | 否 | 创建时间 |
| `updated_at` | `TIMESTAMP` | 是 | 否 | 否 | 更新时间 |
| `deleted_at` | `TIMESTAMP` | 否 | 否 | 是 | 软删除时间 |

约束要求：

- `PRIMARY KEY (id)`
- `UNIQUE (category_code)`
- `UNIQUE (name)`，如后续允许重名可调整

---

## 3.2 文档表 `kb_document`

| 字段名 | 类型 | 必填 | 唯一 | 索引 | 字段说明 |
|---|---|---:|---:|---:|---|
| `id` | `BIGSERIAL` | 是 | 是 | 是 | 文档主键 |
| `document_uid` | `UUID` | 是 | 是 | 是 | 对外稳定文档标识 |
| `category_id` | `BIGINT` | 是 | 否 | 是 | 所属分类 ID |
| `title` | `VARCHAR(256)` | 是 | 否 | 是 | 文档标题 |
| `source_type` | `VARCHAR(32)` | 是 | 否 | 是 | 数据来源类型，第一阶段固定为 `pdf` |
| `file_name` | `VARCHAR(256)` | 是 | 否 | 否 | 原始文件名 |
| `storage_uri` | `VARCHAR(1024)` | 是 | 否 | 否 | 文件存储地址 |
| `mime_type` | `VARCHAR(128)` | 是 | 否 | 否 | 文件 MIME 类型 |
| `file_size` | `BIGINT` | 是 | 否 | 否 | 文件大小，单位字节 |
| `file_sha256` | `CHAR(64)` | 是 | 否 | 是 | 文件摘要 |
| `parse_status` | `VARCHAR(32)` | 是 | 否 | 是 | 解析状态 |
| `vector_status` | `VARCHAR(32)` | 是 | 否 | 是 | 向量状态 |
| `version` | `INT` | 是 | 否 | 是 | 文档版本号 |
| `chunk_count` | `INT` | 是 | 否 | 否 | 当前有效切片数 |
| `last_error` | `TEXT` | 否 | 否 | 否 | 最近一次处理失败信息 |
| `created_at` | `TIMESTAMP` | 是 | 否 | 否 | 创建时间 |
| `updated_at` | `TIMESTAMP` | 是 | 否 | 否 | 更新时间 |
| `deleted_at` | `TIMESTAMP` | 否 | 否 | 是 | 软删除时间 |

约束要求：

- `PRIMARY KEY (id)`
- `UNIQUE (document_uid)`
- `FOREIGN KEY (category_id) REFERENCES kb_category(id)`

索引建议：

- `INDEX idx_document_category_id (category_id)`
- `INDEX idx_document_status (parse_status, vector_status)`
- `INDEX idx_document_file_sha256 (file_sha256)`
- `INDEX idx_document_deleted_at (deleted_at)`

---

## 3.3 切片表 `kb_chunk`

| 字段名 | 类型 | 必填 | 唯一 | 索引 | 字段说明 |
|---|---|---:|---:|---:|---|
| `id` | `BIGSERIAL` | 是 | 是 | 是 | 切片主键 |
| `chunk_uid` | `UUID` | 是 | 是 | 是 | 对外稳定切片标识 |
| `document_id` | `BIGINT` | 是 | 否 | 是 | 所属文档 ID |
| `chunk_no` | `INT` | 是 | 否 | 是 | 文档内切片序号 |
| `page_no` | `INT` | 否 | 否 | 是 | 来源页码 |
| `char_start` | `INT` | 否 | 否 | 否 | 原文起始偏移 |
| `char_end` | `INT` | 否 | 否 | 否 | 原文结束偏移 |
| `token_count` | `INT` | 否 | 否 | 否 | Token 数量 |
| `content` | `TEXT` | 是 | 否 | 否 | 切片原文内容 |
| `content_hash` | `CHAR(64)` | 是 | 否 | 是 | 切片内容摘要 |
| `embedding_model` | `VARCHAR(128)` | 是 | 否 | 是 | 嵌入模型名称 |
| `vector_version` | `INT` | 是 | 否 | 是 | 向量版本号 |
| `vector_status` | `VARCHAR(32)` | 是 | 否 | 是 | 向量状态 |
| `metadata_json` | `JSONB` | 否 | 否 | 否 | 扩展元数据 |
| `created_at` | `TIMESTAMP` | 是 | 否 | 否 | 创建时间 |
| `updated_at` | `TIMESTAMP` | 是 | 否 | 否 | 更新时间 |
| `deleted_at` | `TIMESTAMP` | 否 | 否 | 是 | 软删除时间 |

约束要求：

- `PRIMARY KEY (id)`
- `UNIQUE (chunk_uid)`
- `UNIQUE (document_id, chunk_no)`
- `FOREIGN KEY (document_id) REFERENCES kb_document(id)`

索引建议：

- `INDEX idx_chunk_document_id (document_id)`
- `INDEX idx_chunk_vector_status (vector_status, vector_version)`
- `INDEX idx_chunk_content_hash (content_hash)`
- `INDEX idx_chunk_deleted_at (deleted_at)`

## 4. 表关系设计

### 4.1 分类与文档

关系定义：

- 一个分类可关联多个文档
- 一个文档仅归属于一个分类

关系类型：

- `kb_category` `1:N` `kb_document`

---

### 4.2 文档与切片

关系定义：

- 一个文档包含多个切片
- 一个切片仅属于一个文档

关系类型：

- `kb_document` `1:N` `kb_chunk`

---

### 4.3 切片与向量记录

关系定义：

- 一个切片对应一条当前有效向量记录
- 一条向量记录对应一个切片

关系类型：

- 逻辑关系为 `1:1`
- 物理存储于 Milvus Collection

---

## 5. Milvus Collection 设计

## 5.1 Collection 策略

第一阶段统一采用单 Collection 设计。

推荐 Collection 名称：

- `kb_chunk_vector_v1`

Collection 粒度要求：

- 以切片为最小向量单元
- 全部文档切片统一写入同一个 Collection

---

## 5.2 Collection 字段设计

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `chunk_id` | `Int64` | 是 | 主键，对应关系库 `kb_chunk.id` |
| `document_id` | `Int64` | 是 | 所属文档 ID |
| `category_id` | `Int64` | 是 | 所属分类 ID |
| `dense_vector` | `FloatVector` | 是 | 稠密向量字段 |
| `sparse_vector` | `SparseFloatVector` | 是 | 稀疏向量字段 |
| `embedding_model` | `VarChar` | 是 | 向量模型名称 |
| `vector_version` | `Int32` | 是 | 向量版本号 |
| `created_at` | `Int64` | 否 | 创建时间戳 |

字段要求：

- 主键字段与关系库切片主键保持一致
- 向量记录中保留文档与分类标识，供检索过滤和级联删除使用
- Milvus 中不保存完整业务主数据

---

## 5.3 向量组织方式

第一阶段每条向量记录同时保存：

- 一组稠密向量
- 一组稀疏向量

检索策略支持：

- 稠密检索
- 稀疏检索
- 混合检索

向量版本要求：

- 使用 `vector_version` 标记当前向量版本
- 文档重建或模型升级时，通过版本字段控制向量切换

---

## 5.4 关系库与 Milvus 的配合方式

### 5.4.1 关系型数据库负责

- 分类管理
- 文档元数据管理
- 切片原文存储
- 状态管理
- 版本管理
- 业务删除标记

### 5.4.2 Milvus 负责

- 切片向量存储
- 向量检索
- 基于 `document_id`、`category_id` 的过滤

### 5.4.3 检索结果回查链路

标准流程：

1. 生成查询向量
2. 调用 Milvus 检索
3. 获取命中的 `chunk_id`
4. 回查 `kb_chunk`
5. 关联 `kb_document`
6. 关联 `kb_category`
7. 返回最终业务结果

## 6. 第一阶段业务流程设计

## 6.1 分类管理

支持操作：

- 新增分类
- 查询分类
- 更新分类
- 删除分类
- 启用分类
- 停用分类

数据落点：

- `kb_category`

---

## 6.2 PDF 导入

流程定义：

1. 接收 PDF 文件
2. 保存原始文件到文件存储
3. 创建 `kb_document` 记录
4. 执行 PDF 解析
5. 生成切片
6. 批量写入 `kb_chunk`
7. 生成稀疏向量与稠密向量
8. 批量写入 Milvus
9. 更新文档解析状态与向量状态

状态要求：

- 文档创建后初始化 `parse_status=pending`
- 解析中设置 `parse_status=processing`
- 解析成功设置 `parse_status=success`
- 向量构建成功设置 `vector_status=ready`
- 失败时写入 `last_error`

---

## 6.3 文档查询

支持查询维度：

- 按分类查询
- 按文档 ID 查询
- 按文档 UID 查询
- 按标题查询
- 按状态查询
- 按时间范围查询

查询结果组成：

- 文档基础信息
- 文件信息
- 分类信息
- 切片数量
- 状态信息

---

## 6.4 向量检索

检索输入：

- 查询文本
- 分类过滤条件
- 文档过滤条件
- TopK
- 检索策略参数

检索输出：

- `chunk_id`
- 相关度分数
- 切片内容
- 文档标题
- 分类名称
- 页码
- 文档标识

执行链路：

1. 文本向量化
2. Milvus 检索
3. 业务库回查
4. 结果拼装

---

## 6.5 文档删除

删除策略：

- 第一阶段由服务层统一编排删除
- 支持软删除
- 向量删除与业务删除保持最终一致

标准流程：

1. 读取目标文档
2. 获取文档下全部切片 ID
3. 删除 Milvus 中对应向量记录
4. 软删除或删除 `kb_chunk`
5. 软删除或删除 `kb_document`
6. 清理文件存储中的原始 PDF

幂等要求：

- 删除操作允许重复执行
- 已删除对象再次删除时返回一致结果

---

## 6.6 文档更新

第一阶段统一采用整篇重建策略。

更新流程：

1. 上传新版本 PDF
2. 文档版本号递增
3. 重新解析全文
4. 重新生成全部切片
5. 重新生成全部向量
6. 替换旧版本向量与切片
7. 更新文档状态

版本控制要求：

- `kb_document.version` 持续递增
- `kb_chunk.vector_version` 与文档当前版本保持一致
- 检索默认仅返回当前有效版本数据

## 10. 最终方案摘要

第一阶段数据库设计统一采用以下结构：

- 分类：`kb_category`
- 文档：`kb_document`
- 切片：`kb_chunk`
- 向量：Milvus Collection `kb_chunk_vector_v1`

关系定义统一为：

- 分类 `1:N` 文档
- 文档 `1:N` 切片
- 切片 `1:1` 当前有效向量记录

整体存储架构统一为：

- PostgreSQL 存业务主数据
- Milvus 存向量索引
- 文件存储保存 PDF 原文件

该方案作为知识库 MCP Server 第一阶段数据库设计基线，直接用于后续 DDL、ORM 模型、Repository、Service 与 MCP Tool 开发。
