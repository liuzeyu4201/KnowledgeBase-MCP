# 知识库 MCP Server 数据库设计方案

## 1. 文档目的

本文档以当前代码实现和已确认的后续标准方案为准，定义知识库 MCP Server 的数据库与存储设计。

当前实现与后续标准方案采用四层存储分工：

- PostgreSQL：业务主库，保存分类、文档、切片与状态信息
- Milvus：检索索引库，保存切片级稠密向量与 BM25 稀疏索引
- 文件暂存层：通过 FastAPI 上传接口接收原始文件，并创建暂存文件记录
- 文件存储：保存原始文档与暂存文件，底层可使用对象存储或本地文件系统

当前已经落地的核心能力：

- 分类新增、查询、分页、更新、删除
- PDF / Word / Markdown 文档导入
- 文档详情查询、列表查询、更新、删除
- 切片存储
- Milvus 语义检索、BM25 检索、混合检索
- 跨 PostgreSQL / Milvus / 文件存储的一致性补偿

当前已经确认并准备继续落地的能力：

- 远端大文件上传暂存子系统
- 基于 `staged_file_id` 的文档导入与更新
- 基于 `staged_file_id` 的批量文档导入异步任务
- 任务提交、取消、状态查询
- 基于 PostgreSQL 的任务调度与状态持久化
- 优先级调度
- 协作式取消

## 2. 当前实体拆分

### 2.1 分类实体 `Category`

用于管理文档归属分类。

关键职责：

- 提供稳定分类编码 `category_code`
- 维护分类名称、描述、状态
- 作为文档归属的上级实体

---

### 2.2 暂存文件实体 `StagedFile`

用于表示已上传但尚未或正在被导入的受控文件，是远端上传链路和 MCP 导入链路之间的桥接实体。

关键职责：

- 记录上传文件的元数据、存储位置、摘要和生命周期状态
- 为 MCP Tool 提供稳定的 `staged_file_id`
- 支持过期清理、消费追踪、失败恢复和审计扩展

---

### 2.3 文档实体 `Document`

用于表示一份已导入知识库的文档，是文档生命周期的核心主实体。

关键职责：

- 保存文件信息、存储地址、摘要和来源类型
- 保存解析状态与向量状态
- 维护文档版本号
- 作为切片和向量的上游主实体

---

### 2.4 切片实体 `Chunk`

用于保存文档切片后的内容片段，是检索回查的最小业务单元。

关键职责：

- 保存切片内容、页码、偏移量
- 保存切片对应的向量元数据
- 作为 Milvus 向量记录的主键映射源

---

### 2.5 向量逻辑实体 `ChunkVector`

向量实体在逻辑上存在，物理上保存在 Milvus。

当前实现特点：

- 一条 Milvus 记录对应一个 `Chunk`
- `chunk_id` 与 PostgreSQL `kb_chunk.id` 一一对应
- 稠密向量由 Embedding 模型生成
- 稀疏向量不由应用显式写入，而是由 Milvus 基于 `content` 和 BM25 Function 自动生成

---

## 3. PostgreSQL 表设计

## 3.1 分类表 `kb_category`

| 字段名 | 类型 | 必填 | 唯一 | 索引 | 字段说明 |
|---|---|---:|---:|---:|---|
| `id` | `BIGSERIAL` | 是 | 是 | 是 | 分类主键 |
| `category_code` | `VARCHAR(64)` | 是 | 是 | 是 | 分类编码 |
| `name` | `VARCHAR(128)` | 是 | 是 | 是 | 分类名称 |
| `description` | `VARCHAR(512)` | 否 | 否 | 否 | 分类描述 |
| `status` | `SMALLINT` | 是 | 否 | 是 | 分类状态，`1` 启用，`0` 停用 |
| `created_at` | `TIMESTAMP` | 是 | 否 | 否 | 创建时间 |
| `updated_at` | `TIMESTAMP` | 是 | 否 | 否 | 更新时间 |
| `deleted_at` | `TIMESTAMP` | 否 | 否 | 是 | 软删除时间 |

## 3.2 暂存文件表 `kb_staged_file`

| 字段名 | 类型 | 必填 | 唯一 | 索引 | 字段说明 |
|---|---|---:|---:|---:|---|
| `id` | `BIGSERIAL` | 是 | 是 | 是 | 暂存文件主键 |
| `staged_file_uid` | `VARCHAR(36)` | 是 | 是 | 是 | 稳定暂存文件标识 |
| `status` | `VARCHAR(32)` | 是 | 否 | 是 | 暂存文件状态 |
| `storage_backend` | `VARCHAR(32)` | 是 | 否 | 是 | 存储后端，如 `local`、`s3`、`minio` |
| `storage_uri` | `VARCHAR(1024)` | 是 | 否 | 否 | 受控存储路径或对象存储键 |
| `file_name` | `VARCHAR(256)` | 是 | 否 | 否 | 原始文件名 |
| `mime_type` | `VARCHAR(128)` | 是 | 否 | 是 | MIME 类型 |
| `source_type` | `VARCHAR(32)` | 是 | 否 | 是 | 文档来源类型，如 `pdf`、`docx`、`md` |
| `file_size` | `BIGINT` | 是 | 否 | 否 | 文件大小，字节 |
| `file_sha256` | `VARCHAR(64)` | 是 | 否 | 是 | 文件摘要 |
| `upload_completed_at` | `TIMESTAMP` | 否 | 否 | 否 | 上传完成时间 |
| `expires_at` | `TIMESTAMP` | 否 | 否 | 是 | 过期时间 |
| `consumed_at` | `TIMESTAMP` | 否 | 否 | 是 | 被导入消费时间 |
| `last_error` | `TEXT` | 否 | 否 | 否 | 最近一次错误摘要 |
| `linked_document_id` | `BIGINT` | 否 | 否 | 是 | 成功消费后的文档 ID |
| `linked_task_id` | `BIGINT` | 否 | 否 | 是 | 关联的批量任务 ID |
| `created_at` | `TIMESTAMP` | 是 | 否 | 否 | 创建时间 |
| `updated_at` | `TIMESTAMP` | 是 | 否 | 否 | 更新时间 |
| `deleted_at` | `TIMESTAMP` | 否 | 否 | 是 | 删除时间 |

推荐约束：

- `PRIMARY KEY (id)`
- `UNIQUE (staged_file_uid)`
- `FOREIGN KEY (linked_document_id) REFERENCES kb_document(id)`
- `FOREIGN KEY (linked_task_id) REFERENCES kb_import_task(id)`
- `INDEX idx_kb_staged_file_status (status, created_at DESC)`
- `INDEX idx_kb_staged_file_expires_at (expires_at)`
- `INDEX idx_kb_staged_file_file_sha256 (file_sha256)`
- `INDEX idx_kb_staged_file_mime_type (mime_type)`
- `INDEX idx_kb_staged_file_deleted_at (deleted_at)`

状态建议：

- `uploaded`
- `consuming`
- `consumed`
- `expired`
- `deleted`
- `failed`

说明：

- 该表不表示最终知识文档，只表示“已上传的待处理文件”
- 上传成功后必须同时具备物理文件和数据库记录
- 未被引用的暂存文件可通过 TTL 和后台清理任务回收

## 4. 表关系设计

### 4.1 暂存文件与文档

- 一个暂存文件至多消费成一个最终文档
- 一个文档的当前有效版本对应一个最近消费的暂存文件
- 关系类型：`kb_staged_file 1:0..1 kb_document`

---

### 4.2 分类与文档

- 一个分类对应多个文档
- 一个文档只属于一个分类
- 关系类型：`kb_category 1:N kb_document`

---

### 4.3 暂存文件与批量任务子项

- 一个暂存文件可以被一个批量任务子项引用
- 一个批量任务子项必须引用一个暂存文件
- 关系类型：`kb_staged_file 1:N kb_import_task_item`

---

### 4.4 文档与切片

- 一个文档对应多个切片
- 一个切片只属于一个文档
- 关系类型：`kb_document 1:N kb_chunk`

---

### 4.5 切片与向量记录

- 一个切片对应一条当前有效 Milvus 记录
- 关系类型是逻辑上的 `1:1`
- 物理存储位置是 Milvus Collection

---

## 5. Milvus Collection 设计

## 5.1 Collection 策略

当前实现统一使用单 Collection：

- `kb_chunk_vector_v1`

设计特点：

- 以切片为最小检索单元
- 所有文档切片写入同一 Collection
- 通过 `category_id` 和 `document_id` 进行过滤

---

## 5.2 当前字段设计

| 字段名 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `chunk_id` | `INT64` | 是 | 主键，对应 `kb_chunk.id` |
| `document_id` | `INT64` | 是 | 所属文档 ID |
| `category_id` | `INT64` | 是 | 所属分类 ID |
| `chunk_no` | `INT32` | 是 | 切片序号 |
| `page_no` | `INT32` | 是 | 页码，空值写入 `0` |
| `content` | `VARCHAR(65535)` | 是 | 原始切片文本，同时作为 BM25 输入字段 |
| `dense_vector` | `FLOAT_VECTOR` | 是 | 稠密向量字段 |
| `sparse_vector` | `SPARSE_FLOAT_VECTOR` | 是 | 由 Milvus BM25 Function 自动生成 |

与早期设计相比，当前代码没有在 Milvus 中显式保存以下字段：

- `chunk_uid`
- `embedding_model`
- `vector_version`
- `created_at`

这些信息由 PostgreSQL 负责管理。

---

## 5.3 分析器与索引

### 5.3.1 `content` 分析器

当前默认分析器为自动语言识别模式：

- `language_identifier`
- 识别器：`whatlang`

分词策略：

- 英文：`standard + lowercase`
- 中文：`jieba + removepunct`
- 默认回退：`standard + lowercase`

`content` 字段同时开启：

- `enable_analyzer=True`
- `enable_match=True`

---

### 5.3.2 BM25 Function

当前通过 Milvus 内建函数自动生成稀疏向量：

- 输入字段：`content`
- 输出字段：`sparse_vector`
- 类型：`FunctionType.BM25`

结论：

- 应用层不会自己构造 `sparse_vector`
- 稀疏检索完全由 Milvus BM25 负责

---

### 5.3.3 索引设计

当前索引配置来自运行配置：

- 稠密向量索引：
  - 字段：`dense_vector`
  - 默认索引类型：`AUTOINDEX`
  - 默认度量：`COSINE`
- 稀疏向量索引：
  - 字段：`sparse_vector`
  - 默认索引类型：`SPARSE_INVERTED_INDEX`
  - 度量：`BM25`

默认稠密向量维度：

- `1024`

该值由 `KNOWLEDGEBASE_EMBEDDING_DIMENSION` 控制。

## 6. 关系库与 Milvus 的协作方式

### 6.1 文件上传与暂存链路

远端部署下，标准文件接入流程如下：

1. 客户端通过 FastAPI 上传接口以 `multipart/form-data` 发送原始文件
2. 服务端边接收边写入对象存储或本地受控目录
3. 计算文件大小、摘要、来源类型
4. 在 `kb_staged_file` 创建暂存文件记录，状态写为 `uploaded`
5. 返回 `staged_file_id` 与文件元数据

设计约束：

- 上传接口不负责切片、向量化和文档入库
- 上传成功不等于文档导入成功
- 远端标准路径不再通过 MCP Tool 直接传输大体积 `base64`

---

### 6.2 文档导入写入链路

文档导入流程如下：

1. MCP Tool 接收 `staged_file_id` 和业务参数
2. 锁定 `kb_staged_file`，校验其状态为 `uploaded`
3. 在 `kb_document` 创建文档记录，状态设为 `processing/indexing`
4. 根据 `mime_type` 解析原始文件
5. 按既定策略进行切片
6. 在 `kb_chunk` 批量写入切片
7. 生成稠密向量
8. 向 Milvus 写入切片记录
9. 由 Milvus 自动为 `content` 生成 BM25 稀疏索引
10. 更新 `kb_document` 状态为 `success/ready`
11. 将 `kb_staged_file` 标记为 `consumed` 并记录 `linked_document_id`

---

### 6.3 检索链路

检索流程如下：

1. 接收查询文本
2. 根据 `alpha` 决定检索模式
3. 调用 Milvus 检索切片
4. 取回 `chunk_id`
5. 回查 PostgreSQL 中的 `kb_chunk`、`kb_document`、`kb_category`
6. 仅返回未删除且文档状态为 `success/ready` 的结果

当前检索模式：

- `alpha <= 0.0`：纯语义检索
- `alpha >= 1.0`：纯 BM25 检索
- `0.0 < alpha < 1.0`：混合检索

---

### 6.4 删除与更新链路

当前实现重点保证跨存储一致性。

删除链路：

1. 读取文档与切片
2. 备份旧 Milvus 记录
3. 将原始文件移动到暂存路径
4. 删除 Milvus 记录
5. 软删除 `kb_chunk`
6. 软删除 `kb_document`
7. 数据库提交成功后，再最终删除暂存文件

更新链路：

1. 支持仅更新元数据
2. 替换源文件时，标准路径为先上传新文件生成 `staged_file_id`
3. 更新接口锁定 `kb_staged_file`，采用整篇重建
4. 先暂存旧文件与旧 Milvus 记录
5. 写入新文件并重建文档内容
6. 删除旧 Milvus 记录
7. 物理删除旧切片
8. 重建新切片与新向量
9. 更新文档版本号与状态
10. 将新 `kb_staged_file` 标记为 `consumed`
11. 提交成功后，再最终删除旧文件

---

### 6.5 批量导入异步任务链路

推荐异步执行架构：

- PostgreSQL 任务表作为唯一真相源
- 独立 worker 进程轮询 PostgreSQL 获取任务
- 不引入 Celery、Redis、RabbitMQ 作为第一版前置依赖

任务提交流程：

1. 调用方先上传批量文档，得到一组 `staged_file_id`
2. 在同一事务内写入 `kb_import_task`
3. 在同一事务内批量写入 `kb_import_task_item`
4. 子任务记录中保存 `staged_file_id`
5. 事务提交成功后，任务即处于可调度状态

任务调度流程：

1. worker 按优先级从 `kb_import_task` 选择任务
2. 使用 `FOR UPDATE SKIP LOCKED` 抢占任务
3. 写入 `lease_token`、`lease_expires_at`、`heartbeat_at`
4. 将任务状态更新为 `running`
5. 再按顺序处理 `kb_import_task_item`
6. 每处理一个子任务后更新聚合进度与计数
7. 全部完成后更新任务终态

任务取消流程：

1. 调用取消接口
2. 将 `cancel_requested=true`
3. 若任务尚未开始，worker 抢到后直接标记 `canceled`
4. 若任务正在执行，worker 在协作式检查点主动停止后续执行
5. 已完成的子任务不回滚，未执行的子任务标记为 `canceled`

协作式取消检查点建议：

- 每次开始处理新的子任务前
- 单文档文件落地前
- 单文档解析后
- 单文档切片后
- 单文档写 Milvus 前
- 单文档成功提交后

---

## 7. 一致性补偿设计

当前代码没有使用分布式事务，而是使用“数据库事务 + 外部副作用补偿”。

### 7.1 上传暂存补偿

上传阶段失败时，执行以下回滚：

- 若文件已写入存储但数据库记录创建失败，则删除物理文件
- 若数据库事务回滚，则不得保留 `kb_staged_file` 脏记录
- 若文件大小、摘要或类型校验失败，则整体视为上传失败

---

### 7.2 导入补偿

任一步失败时，执行以下回滚：

- 回滚 PostgreSQL 事务
- 删除已插入 Milvus 的向量记录
- 保持 `kb_staged_file` 仍可重试，或标记为 `failed`

---

### 7.3 删除补偿

任一步失败时，执行以下回滚：

- 回滚 PostgreSQL 事务
- 恢复已删除的 Milvus 记录
- 恢复暂存的原始文件

---

### 7.4 更新补偿

任一步失败时，执行以下回滚：

- 回滚 PostgreSQL 事务
- 删除新写入的 Milvus 记录
- 恢复旧 Milvus 记录
- 删除新文件
- 恢复旧文件

---

### 7.5 异步任务一致性设计

异步任务链路必须满足以下原则：

- PostgreSQL 是任务状态唯一真相源
- 队列状态不独立存在于内存
- worker 是否持有任务，以数据库租约为准
- 任务提交成功的判定标准是数据库事务提交成功

一致性控制要点：

1. 提交任务时，`kb_import_task` 与 `kb_import_task_item` 必须在同一事务中写入
2. worker 抢任务时必须使用行锁，避免多 worker 重复执行
3. worker 启动执行后必须写租约和心跳，防止僵尸任务长期占用
4. worker 崩溃后，其他 worker 可在租约过期后重新接管任务
5. 单个文档导入仍复用现有同步导入逻辑，继续保留 PostgreSQL / Milvus / 文件存储补偿能力
6. 任务主状态由子任务聚合得出，不允许 worker 直接凭内存推断最终结果

防止状态漂移的核心约束：

- 不允许“数据库已提交但队列未入队”的双写模式
- 不允许“内存中认为在运行，但数据库仍是 pending”的状态漂移
- 不允许 worker 完成后不回写聚合进度

---

### 7.6 幂等性设计

异步批量导入需要同时保证任务级和子任务级幂等。

任务级幂等：

- 允许调用方传入 `idempotency_key`
- 若同一 `idempotency_key` 的任务已存在，则直接返回已有任务

子任务级幂等：

- `UNIQUE (task_id, item_no)` 防止重复创建子任务
- 同一个子任务重复被 worker 接管时，通过 `lease_token + status` 控制只能有一个执行者生效

文档导入级幂等：

- 单文档执行仍由现有导入服务负责
- 如果后续需要加强，可额外引入 `(category_id, file_sha256)` 级别的业务去重策略

---

### 7.7 并发安全设计

任务调度必须支持多 worker 并发运行。

关键策略：

- 抢任务使用 `FOR UPDATE SKIP LOCKED`
- 抢到任务后立刻更新状态和租约
- worker 续租时校验 `lease_token`，防止过期 worker 覆盖新 worker 状态
- 聚合更新计数时使用数据库原子更新，不依赖本地内存计数

并发调度顺序建议：

1. `priority DESC`
2. `created_at ASC`
3. `id ASC`

---

## 8. 当前实现边界与后续落地方向

当前尚未按标准方案落地的表和能力：

- 权限表
- 审计表
- 多租户表
- 文档多分类关系表
- 向量独立元数据表
- `kb_staged_file`
- 基于 `staged_file_id` 的标准导入接口族
- 配套 FastAPI 上传接口与清理任务

当前设计取舍：

- 以单分类归属为前提
- 检索粒度为 `chunk`
- 向量版本主要保存在 `kb_chunk.vector_version`
- Milvus 仅作为检索索引层，不作为业务主库
- 异步任务系统优先采用 PostgreSQL 持久化队列，而不是 Celery 或内存队列
- 单文档导入逻辑继续复用现有同步导入服务
- 远端标准文件接入路径采用“普通 HTTP 上传 + MCP from_staged 导入”两段式方案

---

## 9. 结论

当前代码实现与数据库设计的对应关系如下：

- PostgreSQL 是业务真相源
- `kb_document` 是文档生命周期核心实体
- `kb_chunk` 是检索回查核心实体
- Milvus 存储切片级索引，BM25 稀疏向量由内建函数自动生成
- 删除、更新、导入均带补偿逻辑，保证 PostgreSQL、Milvus、文件存储之间的最终一致性
- 远端标准上传方案以 `kb_staged_file` 为桥接实体，负责连接文件传输与 MCP 业务导入
- 后续批量导入异步任务采用 PostgreSQL 任务表 + 独立 worker 轮询方案
- 任务系统以数据库为唯一真相源，支持优先级调度、协作式取消、租约续期、崩溃恢复和幂等控制

本文档已经按当前实现修正，可直接作为后续开发与评审基线。
