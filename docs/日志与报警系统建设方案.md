# 知识库 MCP Server 日志与报警系统建设方案

## 1. 文档目的

本文档基于当前项目代码实现，定义知识库 MCP Server 的生产级日志系统与报警系统建设方案，用于指导后续可观测性建设。

本文档覆盖：

- 当前日志系统现状评估
- 生产级日志系统目标
- 日志字段与分层规范
- 指标与报警体系设计
- 分阶段实施计划

---

## 2. 当前现状评估

### 2.1 当前已具备能力

当前项目已经存在少量日志能力，主要集中在以下模块：

- `knowledgebase/web/middleware.py`
  - 为网页与上传接口注入 `request_id`、`trace_id`
  - 记录基础访问日志
- `knowledgebase/web/routes.py`
  - 记录网页聚合接口的少量业务日志与异常日志
- `knowledgebase/integrations/embedding/validator.py`
  - 记录 embedding 启动校验日志

当前项目还具备一部分链路追踪数据字段基础：

- 请求模型中已广泛保留 `request_id`
- 请求模型中已广泛保留 `trace_id`
- 异步任务表中已落库 `request_id`、`trace_id`

### 2.2 当前存在问题

当前实现还不能视为生产级日志系统，主要问题如下：

- 没有统一日志初始化入口
- 没有统一的日志格式和输出规范
- 没有结构化 JSON 日志
- 没有全链路上下文注入机制
- MCP 主链路没有统一接入访问日志
- Worker 基本没有生命周期日志、任务日志和失败日志
- 没有敏感字段脱敏机制
- 没有慢请求日志和审计日志
- 没有 metrics 指标体系
- 没有 Prometheus、Grafana、Alertmanager
- 没有异常聚合与错误报警系统

### 2.3 当前风险

如果继续保持当前状态，生产环境会面临以下问题：

- 请求失败后无法快速定位是 Web、MCP、Worker、Milvus 还是 Embedding 链路的问题
- 批量导入、单文档导入、删除补偿等后台行为难以追踪
- 无法基于错误率、延迟、积压量建立可靠报警
- 无法区分业务异常、系统异常和外部依赖异常
- 故障只能依赖人工看容器标准输出排查

---

## 3. 建设目标

日志与报警系统的建设目标如下：

- 为 Web、MCP、Worker 建立统一日志基础设施
- 统一结构化日志格式，支持集中采集与检索
- 形成稳定的请求追踪与任务追踪能力
- 为关键业务动作建立审计日志
- 为核心依赖建立可观测性和失败告警
- 通过指标体系驱动报警，而不是仅靠文本日志检索
- 为后续接入 OpenTelemetry、Sentry、Loki、Grafana 预留扩展空间

---

## 4. 总体设计原则

### 4.1 基本原则

- 日志负责排障、审计、追踪
- 指标负责聚合、趋势分析、报警
- 异常追踪负责堆栈聚合和错误归因
- 生产环境默认以结构化标准输出为主，不以本地文件为主
- 所有可观测性能力必须覆盖 Web、MCP、Worker 三条主链路

### 4.2 分工原则

- `Log`
  - 记录离散事件
  - 用于排障和审计
- `Metric`
  - 记录数量、耗时、积压、错误率
  - 用于看板和报警
- `Trace`
  - 记录跨模块、跨依赖链路
  - 用于定位慢链路和失败链路

### 4.3 日志输出原则

- 统一输出到 `stdout`
- 使用 JSON 格式
- 所有日志都带统一上下文字段
- 默认不打印敏感信息明文
- 高频链路避免同步阻塞式日志 I/O

## 5. 生产级日志系统设计

## 5.1 模块划分

建议新增以下模块：

```text
knowledgebase/
  app/
    logging.py          日志初始化、格式化、脱敏、处理器组装
    observability.py    可观测性入口装配
  observability/
    context.py          request_id / trace_id / operator / task_id 上下文
    logging.py          JSON formatter、filter、event builder
    metrics.py          指标注册与埋点封装
    alarms.py           告警事件封装
```

### 5.2 初始化入口

建议在以下两个入口最早阶段调用统一日志初始化：

- `knowledgebase/mcp/server.py`
- `knowledgebase/worker/import_worker.py`

目标：

- App 进程和 Worker 进程使用同一套日志格式
- `uvicorn`、业务代码、第三方库日志统一接管
- 在服务真正启动前就能记录配置、自检和失败原因

### 5.3 日志格式

生产环境建议统一采用 JSON 日志，示例：

```json
{
  "ts": "2026-04-14T10:15:21.381Z",
  "level": "INFO",
  "service": "knowledgebase-app",
  "env": "prod",
  "logger": "knowledgebase.document",
  "event": "document.import.completed",
  "message": "document import completed",
  "request_id": "req_123",
  "trace_id": "trace_123",
  "operator": "mcp-agent",
  "task_id": 18,
  "document_id": 201,
  "category_id": 12,
  "duration_ms": 1842.31
}
```

开发环境可保留可读性更高的文本日志，但字段集必须保持一致。

### 5.4 必备字段规范

所有日志统一建议包含以下字段：

| 字段 | 必填 | 说明 |
|---|---:|---|
| `ts` | 是 | UTC 时间戳 |
| `level` | 是 | 日志级别 |
| `service` | 是 | 服务名，如 `knowledgebase-app`、`knowledgebase-worker` |
| `env` | 是 | 环境标识，如 `dev`、`test`、`prod` |
| `logger` | 是 | logger 名称 |
| `event` | 是 | 稳定事件名 |
| `message` | 是 | 人类可读摘要 |
| `request_id` | 否 | 请求标识 |
| `trace_id` | 否 | 链路追踪标识 |
| `operator` | 否 | 调用者，如用户、Agent、系统任务 |
| `task_id` | 否 | 异步任务 ID |
| `document_id` | 否 | 文档 ID |
| `category_id` | 否 | 分类 ID |
| `staged_file_id` | 否 | 暂存文件 ID |
| `duration_ms` | 否 | 耗时 |
| `error_code` | 否 | 统一业务错误码或系统错误码 |
| `exception_type` | 否 | 异常类型 |

### 5.5 上下文注入

建议引入基于 `contextvars` 的统一上下文机制，维护以下字段：

- `request_id`
- `trace_id`
- `operator`
- `task_id`
- `task_type`
- `document_id`
- `category_id`

上下文注入策略：

- Web 请求：中间件注入
- MCP Tool 调用：统一装饰器或注册器注入
- Worker 执行子项：每个 task item 处理前注入
- 外部依赖调用：继承当前上下文

### 5.6 Logger 分层

建议按职责拆分 logger：

- `knowledgebase.access`
- `knowledgebase.web`
- `knowledgebase.mcp`
- `knowledgebase.document`
- `knowledgebase.category`
- `knowledgebase.search`
- `knowledgebase.embedding`
- `knowledgebase.storage`
- `knowledgebase.worker`
- `knowledgebase.audit`
- `knowledgebase.security`

不要继续把所有业务日志都打到 `knowledgebase.web`。

### 5.7 日志分类

建议分成四类日志：

#### 5.7.1 访问日志

用于记录请求摘要：

- HTTP 方法
- 路径
- 状态码
- 耗时
- 请求体大小
- 响应体大小
- 调用方 IP
- request_id
- trace_id

#### 5.7.2 业务日志

用于记录核心业务节点：

- 分类创建、更新、删除
- 文档导入、更新、删除
- 向量生成成功或失败
- Milvus 写入与删除
- 暂存文件上传、消费、过期、清理
- 任务提交、抢占、取消、完成、失败

#### 5.7.3 审计日志

用于记录需要长期追溯的敏感业务操作：

- 删除分类
- 删除文档
- 取消批量任务
- 覆盖更新文档
- 手工清理对象或任务
- 启停关键配置开关

审计日志建议单独 logger：

- `knowledgebase.audit`

并确保字段中包含：

- operator
- object_type
- object_id
- action
- result
- request_id
- trace_id

#### 5.7.4 错误日志

用于记录：

- 未捕获异常
- 外部依赖失败
- 重试与补偿动作
- 数据不一致
- 启动失败

错误日志必须明确区分：

- 业务错误
- 系统错误
- 外部依赖错误

### 5.8 事件命名规范

建议统一使用稳定事件名，避免大量自然语言变体：

- `http.request.completed`
- `http.request.failed`
- `mcp.tool.called`
- `mcp.tool.completed`
- `mcp.tool.failed`
- `category.create.completed`
- `category.delete.completed`
- `document.import.accepted`
- `document.import.started`
- `document.import.completed`
- `document.import.failed`
- `document.update.completed`
- `search.retrieve.completed`
- `embedding.startup.validation_passed`
- `embedding.request.completed`
- `embedding.request.failed`
- `milvus.upsert.completed`
- `milvus.query.failed`
- `worker.task.claimed`
- `worker.task.completed`
- `worker.task.failed`
- `storage.gc.retry_scheduled`

### 5.9 敏感信息脱敏

必须实现日志脱敏，至少覆盖以下字段：

- `api_key`
- `authorization`
- `secret`
- `password`
- `access_key`
- `token`

脱敏原则：

- 默认不打印原值
- 仅允许打印是否已配置
- 必要时只显示前 2 位和后 2 位

### 5.10 日志性能策略

高并发场景下建议采用：

- `QueueHandler + QueueListener`
- 非阻塞式日志处理
- 高频成功日志支持采样
- 对超高频第三方库日志下调级别

建议：

- 成功访问日志全量保留
- 高频内部轮询日志采样
- 心跳日志按时间窗口聚合

## 6. 指标系统设计

## 6.1 设计原则

报警应主要依赖指标，而不是直接扫描原始日志文本。

建议新增 Prometheus 指标出口：

- `GET /metrics`

### 6.2 指标分类

#### 6.2.1 请求指标

- `http_requests_total`
- `http_request_duration_seconds`
- `mcp_tool_requests_total`
- `mcp_tool_duration_seconds`

标签建议：

- `method`
- `path`
- `tool`
- `status_code`
- `result`

#### 6.2.2 业务指标

- `category_operations_total`
- `document_import_total`
- `document_update_total`
- `document_delete_total`
- `search_requests_total`
- `search_result_count`

#### 6.2.3 异步任务指标

- `import_task_submitted_total`
- `import_task_running`
- `import_task_backlog`
- `import_task_item_failures_total`
- `import_task_item_duration_seconds`
- `worker_heartbeat_timestamp`

#### 6.2.4 外部依赖指标

- `embedding_requests_total`
- `embedding_request_duration_seconds`
- `embedding_failures_total`
- `milvus_query_duration_seconds`
- `milvus_write_duration_seconds`
- `milvus_failures_total`

## 7. 报警系统设计

## 7.1 总体架构

建议生产环境采用以下组合：

- Prometheus
  - 抓取 `/metrics`
- Alertmanager
  - 负责告警路由、聚合、抑制、通知
- Grafana
  - 展示监控面板
- Sentry
  - 聚合异常堆栈与错误上下文
- Loki 或 OpenSearch
  - 存储结构化日志

### 7.2 告警来源分层

#### 7.2.1 指标告警

作为主告警通道。

适用于：

- 错误率
- 延迟
- backlog
- worker 存活
- 外部依赖失败率

#### 7.2.2 异常聚合告警

由 Sentry 触发。

适用于：

- 未捕获异常
- 新异常类型出现
- 同类异常短时间激增

#### 7.2.3 日志检索

仅用于事后排障，不作为主报警机制。

---

## 8. 第一版告警规则建议

### 8.1 P1 级告警

触发即需要立即处理。

- App 进程不可用
- `/healthz` 连续失败
- MCP 主接口连续失败
- Worker 心跳丢失超过阈值
- PostgreSQL 不可达
- Milvus 不可达
- Embedding 启动校验失败

### 8.2 P2 级告警

影响主要功能，但系统未完全不可用。

- HTTP 5xx 比例 5 分钟超过 1%
- MCP Tool 错误率持续升高
- 文档导入失败率超过阈值
- Embedding 请求失败率超过 5%
- Milvus 查询失败率超过 5%
- 任务积压量持续增长
- 存储 GC 积压超过阈值

### 8.3 P3 级告警

用于性能退化和风险预警。

- HTTP P95 延迟超过阈值
- MCP Tool P95 延迟超过阈值
- Embedding 平均耗时显著升高
- Worker 单任务耗时异常增大
- 清理补偿任务重试次数异常增多

### 8.4 推荐阈值

第一版可先采用以下保守阈值：

- `http_5xx_rate > 1%` 持续 5 分钟
- `mcp_tool_error_rate > 2%` 持续 5 分钟
- `embedding_error_rate > 5%` 持续 5 分钟
- `worker_heartbeat_missing > 120s`
- `import_task_backlog > 50` 持续 10 分钟
- `storage_gc_backlog > 100` 持续 15 分钟
- `http_p95 > 2s` 持续 10 分钟
- `mcp_tool_p95 > 5s` 持续 10 分钟

## 9. 通知路由设计

建议至少支持以下通知通道：

- 企业微信机器人
- 飞书机器人
- PagerDuty 或 Opsgenie
- 邮件

通知策略建议：

- P1：值班通道 + 即时 IM + 电话升级
- P2：值班通道 + 即时 IM
- P3：IM 群通知或工作时间邮件

应避免：

- 所有异常都直接打断值班人员
- 同一类告警高频重复轰炸

因此 Alertmanager 需要配置：

- 分组聚合
- 抑制规则
- 静默窗口
- 重复发送间隔

## 12. 分阶段实施计划

## 12.1 第一阶段：统一日志基础设施

目标：

- 建立统一日志初始化
- 统一 JSON 日志
- 建立上下文注入
- 覆盖 Web、MCP、Worker 三条链路

实施内容：

- 新增 `knowledgebase/app/logging.py`
- 新增上下文管理模块
- 接管 `uvicorn`、业务 logger
- 为 MCP 增加请求级日志包装
- 为 Worker 增加任务生命周期日志
- 实现敏感字段脱敏

验收标准：

- 任何请求都能带 `request_id` 和 `trace_id`
- 任何任务失败都能定位到 task_id、item_id、document_id
- 生产容器日志统一为 JSON

## 12.2 第二阶段：指标与监控

目标：

- 建立 `/metrics`
- 接入 Prometheus 与 Grafana
- 建立核心告警规则

实施内容：

- 新增 Prometheus 指标注册
- 为 Web、MCP、Worker 埋点
- 配置 Grafana 面板
- 配置 Alertmanager 规则与通知渠道

验收标准：

- 可查看请求量、错误率、耗时、任务积压
- worker 掉线或任务积压可自动报警

## 12.3 第三阶段：异常聚合与链路追踪

目标：

- 接入 Sentry
- 为外部依赖补 trace 信息
- 为慢链路提供更强诊断能力

实施内容：

- 接入 Sentry SDK
- 捕获未处理异常
- 按 request_id / trace_id 补上下文
- 预留 OpenTelemetry 接口

验收标准：

- 关键异常可在 Sentry 聚合
- 能从请求追踪到任务再追踪到外部依赖失败

## 13. 推荐实施优先级

当前项目最优先的不是先上大而全的平台，而是先补基础设施。

推荐优先级如下：

1. 统一日志初始化
2. MCP 主链路日志接入
3. Worker 生命周期日志
4. 敏感字段脱敏
5. Prometheus 指标
6. 核心告警规则
7. Sentry
8. Trace

## 14. 对当前项目的直接建议

结合当前代码结构，建议下一步优先做以下四件事：

### 14.1 统一日志入口

在 App 和 Worker 启动最早阶段统一调用日志初始化，不再依赖默认日志行为。

### 14.2 MCP 主链路补齐上下文日志

当前核心业务大量通过 MCP Tool 进入，但这条链路没有完整访问日志，这是当前最关键的观测缺口。

### 14.3 Worker 任务日志补齐

需要记录：

- 抢占任务
- 开始处理
- 心跳
- 单项成功
- 单项失败
- 重试
- 取消
- 聚合完成

### 14.4 指标驱动报警

不要继续用“看日志有没有 error”作为主要运维手段，应尽快建立：

- backlog
- 错误率
- 延迟
- worker 存活

这四类核心指标。

---

## 15. 总结

当前项目已有局部日志基础，但距离生产级日志与报警系统还有明显差距。

生产级建设的正确路径应为：

- 先统一日志
- 再统一上下文
- 再补指标
- 最后做报警和异常聚合

日志系统和报警系统不应被视为附属功能，而应被视为知识库 MCP Server 的基础设施能力，与 PostgreSQL、Milvus、MinIO、Embedding 接入同等重要。
